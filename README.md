# Сервис аналитики поисковых запросов (wb-search-top)

Я разработал этот высоконагруженный микросервис на Go в качестве тестового задания для стажировки в RWB (Wildberries & Russ). Сервис агрегирует поисковые логи из брокера сообщений, рассчитывает топ популярных запросов за последние 5 минут и предоставляет быстрый HTTP API для отдачи отфильтрованных данных виджету «Сейчас ищут».

---

## Реализованные дополнительные пункты

* **Динамический стоп-лист:** Я реализовал API для управления стоп-листом, что позволяет мне добавлять и удалять нежелательные слова «на лету» без перезапуска сервиса.
* **Мониторинг:** Я внедрил базовые метрики в формате Prometheus для наблюдения за состоянием сервиса, временем агрегации и входящим HTTP-трафиком.
* **DX (Developer Experience):** Я настроил процесс быстрого локального запуска для проверяющих. Мой `docker-compose.yml` одной командой поднимает сам сервис, брокер сообщений (RabbitMQ), базу данных (Redis) и систему сбора метрик (Prometheus).
* **Борьба с накрутками и ботами** Я реализовал защиту от спама одного запроса одним пользователем, что позволяет получать честный топ.

## Инструкция по локальному запуску проекта и примеры запросов к API

Я подготовил чистое окружение на официальных Docker-образах (без использования тяжелых сборок).

**1. Клонирование репозитория**
Склонируйте проект из моего публичного репозитория `https://github.com/ivanSaichkin/wb-search-top` и перейдите в его директорию:

```bash
git clone https://github.com/ivanSaichkin/wb-search-top
cd wb-search-top
```

**2. Подготовка переменных окружения**
Я создал файл `.env` в корневом каталоге проекта для конфигурации портов и доступов:

```env
APP_PORT=8080
REDIS_PORT=6379
RABBITMQ_PORT=5672
RABBITMQ_USER=guest
RABBITMQ_PASSWORD=guest
```

**3. Запуск инфраструктуры**
Контейнеры снабжены healthcheck-проверками, поэтому мое приложение запускается только после полной готовности базы и брокера.

```bash
docker-compose up --build -d
```

**4. Проверка статуса**

```bash
docker ps
```

### Примеры запросов к API

**1. Получение Топ-N популярных запросов**
Я сделал ручку, которая возвращает список трендовых запросов за последние 5 минут, очищенный от стоп-слов.

* **Запрос:** `GET /api/v1/top?limit=3` (если параметр `limit` опущен, я по умолчанию возвращаю 10 элементов).
* **Ответ (`200 OK`):**

    ```json
    [
      {
        "query": "ноутбук",
        "unique_hits": 1050
      },
      {
        "query": "беспроводные наушники",
        "unique_hits": 820
      },
      {
        "query": "смартфон",
        "unique_hits": 430
      }
    ]
    ```

**2. Добавление слова в динамический стоп-лист**

* **Запрос:** `POST /api/v1/stoplist`
* **Тело запроса (JSON):**

    ```json
    {
      "word": "казино"
    }
    ```

* **Ответ:** `201 Created`

**3. Удаление слова из стоп-листа**

* **Запрос:** `DELETE /api/v1/stoplist?word=казино`
* **Ответ:** `200 OK`

## Контракт данных (Брокер)

Поскольку в ТЗ формат payload не был задан, я сам спроектировал контракт для смежной команды поиска. Вот пример JSON-сообщения, которое мой консьюмер ожидает в очереди `search_events`:

```json
{
  "query": "купить игровой ноутбук",
  "user_id": "usr_9921a8b3f",
  "timestamp": "2026-05-26T17:30:00Z"
}
```

### Почему я выбрал именно эти поля

* `query` **(string):** Сам поисковый запрос в сыром виде. Он нужен мне для частотного анализа при фильтрации стоп-слов.
* `user_id` **(string):** Идентификатор пользователя. Я добавил это поле специально для борьбы с накрутками парсеров. Оно позволяет мне считать **уникальные** действия в рамках временного окна, предотвращая искусственный вывод запроса в топ за счет спама от одного источника.
* `timestamp` **(time.Time):** Время совершения поиска на клиенте. Я использую его для защиты от "эффекта отставания". Если RabbitMQ задержит доставку или мой сервис временно упадет, при обработке старых сообщений сервис сопоставит их с реальным историческим окном, а не запишет с текущим временем (`time.Now()`).

## Обоснование архитектуры

* **База данных (Redis):** Я выбрал Redis, так как по условиям задачи нагрузка на чтение в 10–50 раз превышает нагрузку на запись. Хранение данных в оперативной памяти (In-Memory) позволяет мне отдавать топ быстро, гарантируя высокую пропускную способность.
* **Брокер сообщений (RabbitMQ):** Я выбрал RabbitMQ, так как имею хороший опыт работы с ним. Для надежности я отключил `auto-ack` и настроил **Dead Letter Queue (DLQ)**. Если приходит битый JSON или падает БД, я отправляю `Nack`, и лог улетает в DLQ с TTL 7 дней, не блокируя основную очередь.
* **Хранение топа (Sorted Set):** Я храню актуальный топ в `ZSET`. Это идеальная структура для лидербордов: элементы сразу упорядочены по `score` (количеству уникальных хитов).
* **Хранение стоп-листа (Set):** Я храню стоп-слова в структуре Set. Поиск слова в ней выполняется за $O(1)$, что минимизирует накладные расходы при фильтрации.

## Trade-offs (Компромиссы) и бизнес-логика

При проектировании архитектуры я столкнулся с логическими противоречиями продуктовой задачи и пошел на следующие компромиссы ради производительности:

### 1. Скользящее окно vs Фоновая агрегация

**Проблема:** Если вычислять честное окно строго "за последние 5 минут" в момент HTTP-запроса, Redis упадет под огромным потоком тяжелых операций.
**Мое решение (Trade-off):** Я пожертвовал строгой синхронностью до миллисекунды. Я внедрил фоновый воркер `RunAggregatorWorker`, который с заданным интервалом производит пересчет топа в фоне и сохраняет готовый слепок. HTTP-ручка `GetTop` моментально читает уже предрассчитанный результат.

### 2. Pre-фильтрация vs Post-фильтрация (Стоп-лист)

**Проблема:** Если фильтровать поисковые фразы на этапе записи в брокер, то при добавлении нового слова в стоп-лист "на лету", старые нежелательные запросы останутся в топе еще 5 минут. Если делать вычитание прямо в Redis при чтении, это убьет RPS.
**Мое решение (Trade-off):** Я применил **Post-фильтрацию с запасом**. Мой `SearchService` запрашивает из Redis сырой топ с пятикратным запасом (`limit * 5`). Затем прямо в памяти Go я быстро отсекаю запрещенные слова за $O(1)$, пока не наберется ровно `limit` чистых элементов. Это позволяет мне применять стоп-лист мгновенно без деградации базы.

## Борьба с накрутками и ботами (Anti-Fraud)

Чтобы решить эту проблему, я внедрил подсчет **Уникальных Хитов (Unique Hits)**.

Вместо того чтобы делать слепой `INCR` на каждое сообщение, я привязываю событие к паре `user_id` + `query`. Для агрегации в пределах минуты я использую структуру `HyperLogLog` (через `PFADD` в Redis). Если бот пришлет 50 000 одинаковых запросов `"айфон"` с одного ID за минуту, вес этого запроса в моем рейтинге увеличится **всего на 1**. Накрутка полностью изолируется и больше не ломает статистику виджета.
